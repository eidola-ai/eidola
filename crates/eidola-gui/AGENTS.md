# crates/eidola-gui — gpui-based Eidola

Guidance for AI coding agents working on the gpui macOS app. Cross-cutting workspace context (server, app-core, build-system, conventions) lives in the top-level `AGENTS.md`.

## What this app is

A native Rust client for Eidola, built on [gpui](https://github.com/zed-industries/zed/tree/main/crates/gpui) (the immediate-mode UI framework Zed is built on) and [gpui-component](https://github.com/longbridge/gpui-component) (a shadcn-style widget library on top of gpui). The sole macOS GUI for the project; the CLI in `crates/eidola-cli/` shares the same `crates/eidola-app-core/` backend. macOS-only today; Linux is the next target.

**State & async doctrine — read first.** All state ownership, async tasks, and cross-window synchronization follow `docs/architecture/state.md` (domain stores, `Loadable`, task-as-field with replace-cancels, the invalidation bus, `WindowInput`). Where existing code in this crate contradicts the doctrine, the doctrine wins — the migration is in progress; do not extend the old patterns (`Core::spawn`/busy flag, detached tasks, view-owned domain data).

## Stores (`src/stores/`, `src/loadable.rs`, `src/bridge.rs`)

`AppCore` lives in `crates/eidola-app-core/`; gpui can't host it directly because gpui's executor is smol-based and `AppCore` has its own tokio multi-thread runtime. State ownership and the async bridge follow `docs/architecture/state.md` (the binding doctrine) — read it before touching state-bearing code. The old `Core` god-object has been dissolved into per-domain stores.

- **Domain stores** (`src/stores/`): one gpui entity per domain — `ConfigStore`, `ModelsStore`, `AccountStore`, `WalletStore`, `SpacesStore`, `UpdateStore` — created once at startup, bundled in `Stores` (held by `AppGlobal`), and handed to every view's constructor. Each store owns its `Loadable` snapshots, its in-flight `Task` *fields* (one slot per operation, replace-cancels), its subscription to the invalidation bus, and *all* mutations of its domain. **No shared busy flag.** Views call store methods and observe the specific store entities they render; they never call `AppCore` directly.
- **`Loadable<T>`** (`src/loadable.rs`): the universal async cell — `NotLoaded` / `Loading` / `Loaded { value, stale }` / `Failed { error, prior }`. A re-fetch over existing data stays `Loaded { stale: true }` (never a blank `Loading` flash); `Loading` is for initial loads only and always implies a live task in the sibling slot.
- **Method shapes:** `refresh_*` is fire-and-notify (the store owns the task; callers observe). `request_*` is awaitable (returns a `oneshot::Receiver`; the *caller* awaits it inside its own slot, so the work dies with the caller — e.g. the onboarding checkout poll). Ownership = cancellation authority.
- **The bus bridge** (`stores::install_bus_bridge`, installed once at startup): a task on `AppCore::runtime()` forwards every `AppCore::subscribe_changes()` event through an mpsc channel into one gpui main-thread loop, which dispatches each `Change` to the owning store(s) (`Change::Config` → config + models, `Change::Wallet` → wallet, …). On `Lagged` it refreshes everything. This is the **only** place tokio receivers touch gpui, and its app-lifetime task is the **one sanctioned `.detach()`** (commented as such) — every other task is an entity-owned field.
- **`src/bridge.rs`**: the runtime-agnostic `oneshot`/`mpsc` glue. `bridge(core, f)` runs `f` on the tokio runtime and awaits the result on gpui (used inside store `cx.spawn` tasks). The free functions `chat_stream` / `get_space_messages` now back the **`Space` entity**'s owned runner/load tasks (step 3); `list_attestations` / … still back the Record reader's view-owned tasks (a window-scoped reader entity is a later step). All take the `Arc<AppCore>` out of the `Stores` bundle and hand back a receiver the owning entity awaits inside its own `Task` slot.
- **Stub mode (tests):** `Stores::stub_with(StoresStub { … }, cx)` builds backend-less stores from a declarative fixture (replacing `Core::stub()` field-poking). Each store also has a `stub(...)` constructor. With no backend, `refresh_*`/`request_*` are no-ops, so views that drive async work transition their *local* state and stop at the backend guard — exactly the behavior tests assert. `Stores::for_test(Arc<AppCore>, cx)` injects a real tempdir-backed `AppCore` for store-machinery tests (`tests/stores.rs`).

## View structure

Every view takes the `Stores` bundle (or a single store entity from it) at construction and observes the specific stores it renders; the "Stores" column lists those data sources.

| File | Window root | Stores (data sources) | Purpose |
|---|---|---|---|
| `space.rs` | — (`Space` domain entity) | `SpacesStore` registry | The shared per-conversation entity: transcript, streaming, submit runner, per-space model selection (see [The Space entity](#the-space-entity)) |
| `chat.rs` | `ChatView` | `ConfigStore`, `ModelsStore`, `AccountStore`, `WalletStore`, plus the shared `Entity<Space>` from `SpacesStore` | Main chat window — a **window-local lens** over a `Space` (see [Chat — the ChatView lens](#chat--the-chatview-lens)), plus the onboarding empty states (see [Onboarding](#onboarding--the-chat-windows-empty-states)) |
| `library.rs` | `LibraryView` | `SpacesStore` (index + optimistic archive + optimistic rename) | Library window — table of contents of past spaces; reopen / archive / inline rename (double-click) |
| `about.rs` | `AboutView` | — | About window singleton — wordmark, version, purpose copy, license note, "View on GitHub" link |
| `updates.rs` | `UpdatesView` | `UpdateStore` | Updates window — the verified update-check state machine (see [Updates window](#updates-window)) |
| `record.rs` | `RecordView` | `Stores::app_core()` via `bridge::{list_attestations,list_requests,spend_trail,attestation_detail,request_detail}` (window-scoped reads; no global store) | The Record window — raw local trail: attestations / requests / spending (see [The Record](#the-record-window--the-trust-doors-engine-room)) |
| `settings.rs` | `SettingsView` | constructs the sub-views below | Settings window — quiet two-pane layout: nav band (General · Account · Wallet) on `theme.sidebar`, selected pane in the content column |
| `general.rs` | `GeneralView` | `ConfigStore` | ...Base URL (honest about override-vs-pin, one-click revert) + ⌥-revealed advanced rows (attestation URL, domain separator, CAs, measurement summary linking to the Record) |
| `account.rs` | `AccountView` | `ConfigStore`, `AccountStore` (`request_checkout`, `create_account`, `reset_account`) | ...balance + pools (humanized expiry), plans/checkout (shared `plans.rs` rows), two-step inline reset confirm |
| `wallet.rs` | `WalletView` | `WalletStore` (`lifecycle` listing + `recover`) | ...credential history with lifecycle states from the local `credential_lifecycle` view (active / in flight / spent / expired) |
| `plans.rs` (no window) | — | — | shared plans-row presentation (`plan_rows`) used by both the chat onboarding plans page and the Account pane; takes an `Rc` select-handler so each caller routes clicks back into its own entity |

All view roots are wrapped in `gpui_component::Root` before being handed to `cx.open_window`. **Root is required** — `gpui_component::Input` calls `Root::read(window, cx)` to track the focused input, and panics if the window's root view type isn't `Root`.

**No shared busy flag.** Each store operation has its own task slot, so concurrent refreshes never starve each other — the structural fix for the wave-2 launch-order bug where a shared `busy` gate dropped the first window's model-list fetch behind the startup wallet recovery (see the replay test in `tests/stores.rs`). Multi-part work either composes inside one store task (e.g. `WalletStore::refresh` fetches lifecycle + active credentials in one `bridge` call) or composes by *event* — write through `AppCore`, let the bus drive each store's own refresh; one store never orchestrates another's tasks. View-owned flows that must outlive a single observe (onboarding account-create / checkout / balance poll) use the awaitable `AccountStore::request_*` methods and own the await in the view's own task slot.

## The Space entity

`space.rs`'s `Space` is the per-conversation **domain entity** (`docs/architecture/state.md` → "Space entities — shared, registried"). It owns everything about one conversation, so a window opening/closing never re-derives or destroys it:

- **State it owns:** `id: Option<String>` (`None` for a blank ⌘N space until its first exchange persists and assigns one), `transcript: Loadable<Vec<ChatMessageView>>`, the live `streaming: Option<StreamingResponse>` buffers (reasoning + content + disclosure-`expanded`), `selected_model` (per-space, not per-window — see the picker note below), and `last_submitted_model` (set before the backend guard so behavior tests see what a real send would use).
- **Tasks-as-fields (no `.detach()`):** `submit_runner: Option<Task<()>>` is the single runner slot — a submit while one is in flight is a no-op (the current UX), and the runner owns both the streaming pump *and* the post-stream transcript reload. `load_task: Option<Task<()>>` is the supersede slot for the reopened-space initial transcript load. ChatView's old detached streaming/load tasks moved here; this is where that detachment debt was paid.
- **The load-vs-submit race is serialized inside the entity**, which retired the old `transcript_generation` counter: one entity owns both the initial load and every submit, so they serialize on `&mut self` between awaits. A submit drops the in-flight `load_task` and `apply_loaded_transcript` is a no-op while `streaming.is_some()` — a slow initial load that finishes after a local submit cannot clobber the just-appended user turn (the submit's own post-stream reload is authoritative). Replay test: `stale_initial_space_load_does_not_replace_submitted_prompt`.
- **Streaming reasoning stays ephemeral.** The DB persists only the assistant's final content; reasoning is captured into the streaming buffer and re-attached to the new last assistant entry at finalize. `merge_messages_from_db` preserves reasoning *by position* across a transcript reload (we only ever append, so positions are stable).
- **`EventEmitter<SpaceEvent>`** — `MessagesChanged` / `StreamDelta` / `StreamEnded` / `Failed(AppError)`. Plain `cx.observe` covers re-render; the events let a view react *semantically* (tail-scroll on `StreamDelta`, degraded-onboarding routing on `Failed`).

**The registry lives in `SpacesStore`** (`HashMap<SpaceId, WeakEntity<Space>>`):

- `open(id)` is **get-or-create with join-existing**: a live registered entity is returned as-is, so two windows opening the same space share **one** `Space` (and its single transcript load). This is the structural fix for wave-2 bug 4 — a submit/stream in window A appears in window B live, because both render the same entity. Replay/behavior tests: `two_windows_on_one_space_share_state` (behavior) and `spaces_registry_joins_existing_and_blanks_are_distinct` (stores).
- `blank()` mints an id-less space for ⌘N (instant, no load). The registry **adopts** it once its first exchange assigns an id — whether that exchange **succeeds** (`StreamEnded`) **or fails after the space was persisted** (`Failed`). `SpacesStore` subscribes to both events, reads the now-present `id()`, and keys the entity under it (the shared `adopt_blank` path; a no-op when the id is still `None`, e.g. a pre-persist `Failed` like `NoAccount`) — so a later `open(that_id)` joins the same entity. The id reaches a *failed* blank because app-core wraps any post-persist error as `AppError::ChatFailed { space_id }`; `Space`'s failure arm reads `chat_space_id()` off the wrapper into `self.id` before emitting `Failed` (with the unwrapped source, via `root()`, so the view's error routing sees the real variant). This closes the former known gap where a blank window's first exchange failing never taught it the persisted id (the id only arrived via `ChatResult` on success). Regression test: `blank_space_adopts_id_on_wrapped_failure`.
- `Change::Space(id)` from the invalidation bus is routed (in `stores::dispatch_change`) to `SpacesStore::notify_space_changed(id)`, which tells the live registered `Space` to refresh its transcript (how an in-process CLI write to the same space appears). The space ignores the refresh while its own submit is streaming.

## Chat — the ChatView lens

`ChatView` is a **window-local lens** over a shared `Space`. It holds `Entity<Space>` plus only window-y state: the composer (`prompt_editor`), the `error` band + `show_plans_after_error` presentation, the onboarding flow flags, `Entity<WindowInput>` / `model_picker_open` (⌥ reveal + picker), and its focus handle. The transcript, streaming, submit, and model selection all live on the `Space` — `ChatView::render_transcript` reads them via `self.space.read(cx)`, and `submit` resolves the model (space selection → config default → fallback) then calls `space.submit(prompt, model, cx)`.

- It `observe`s the `Space` for re-render and `subscribe_in`s for `SpaceEvent`: `StreamDelta`/`MessagesChanged`/`StreamEnded` re-render; `Failed(e)` runs `apply_chat_failure` (the window-local error band + `InsufficientBalance` → degraded plans surface). The streaming state itself is cleared by the `Space`, not the view.
- The **composer draft is intentionally window-local** (not on `Space`): two windows on one space are two cursors with two drafts. The error band is likewise window-local so each window surfaces its own last-submit error.
- The onboarding empty-state machine stays in ChatView (it is windowy), reading ConfigStore/AccountStore/WalletStore and the `Space`'s message/streaming counts to short-circuit to `Ready`. The checkout poll stays view-owned.

## Onboarding — the chat window's empty states

From zero to first answer without leaving the page: onboarding is not a wizard window, it's what the chat window's *empty state* renders until the account is usable. The state machine lives in `ChatView` (`chat.rs`):

- **Stage is derived, not stored.** `ChatView::onboarding_stage(core, composer_empty)` returns `Welcome` (no account in `config_state`), `Plans` (account + *known-zero* balance + empty wallet credentials), or `Ready`. Any page content — messages, an in-flight stream, an error band, or composer text — short-circuits to `Ready` so the onboarding pages only ever replace a genuinely empty page. A `None` balance (not yet fetched) is `Ready`, never `Plans`: we don't claim the user is unfunded on an assumption. The only stored bridge is `OnboardingFlow.entered_plans`, which holds the page on `Plans` between account creation and the first balances fetch so the flow doesn't flash through the blank page.
- **`OnboardingFlow`** holds the local in-flight bits — `creating_account`, `checkout_pending`, `awaiting_checkout`, errors, `dismissed` — and every flag corresponds to a real request currently running or an explicit user choice (the no-fake-states rule). Async work goes through `AccountStore`'s awaitable `request_*` methods (account-create, checkout, balance poll), giving the view typed `AppError` results in its own task slot.
- **Welcome** — wordmark + three sentences + a single "Begin" button → `account_create` (anonymous; nothing to fill in). Success refreshes `config_state`, sets `entered_plans`, and kicks `fetch_plans_data`; failure renders inline with Begin available for retry.
- **Plans** — `prices` rendered as hairline-rule rows (name · price, credits underneath) in the prose column; clicking a row → `account_checkout` → `cx.open_url` → `awaiting_checkout` + a 3-second balance poll (`start_balance_poll`, each tick a real `GET /v1/account/balances`; poll errors surface inline and polling continues). A positive balance flips the derived stage to `Ready`. A quiet "I'll do this later" link sets `dismissed`.
- **Degraded honesty** — a later submit failing with `AppError::InsufficientBalance` (typed routing, no string matching — see `apply_chat_failure`) sets `show_plans_after_error`, which renders the same plans list below the transcript's error band. Not a modal.
- Silent credential provisioning itself is app-core's job (see the workspace `AGENTS.md`): a funded account never sees any of this — `chat()` auto-allocates from balance.

Behavior tests cover the stage derivation and each transition's in-flight state (stub core: handlers early-return after the local mutation, mirroring `submit`); visual snapshots cover `onboarding_welcome`, `onboarding_plans`, `onboarding_plans_waiting`, and `chat_insufficient_balance_plans`.

## Theme — Circadian (`src/theme.rs`)

Two `ThemeConfig`s, "Circadian Day" (Light) and "Circadian Night" (Dark), installed onto the global `gpui_component::Theme` after `gpui_component::init`. Switching is driven by OS appearance:

- `Theme::sync_system_appearance` reads the appearance via `cx.window_appearance()` (or `window.appearance()` if a window is passed) and applies the matching config.
- Each opened window subscribes to appearance changes via `theme::observe_window_appearance(window)` so toggling macOS Light/Dark updates live.

The palettes are anchored on two fixed backgrounds (the product's "good paper at noon, reading lamp at midnight" pair): **Day `#fefaf5`** — every other day surface is the same warm family translated up in lightness — and **Night `#15191e`** — a cool blue-grey dark whose elevated surfaces (cards, chips, borders, rows) all follow the anchor's blue-grey ramp (R < G < B throughout), while text stays warm-grey (`#d4d0c8`) and the brand stays warm orange (`#c39669`). The warm-on-cool night tension is deliberate; don't "fix" it by cooling the foregrounds. An earlier palette was seeded from the marketing site — the site is no longer the reference, these anchors are.

**Body font is Newsreader 16pt** (SIL OFL 1.1) — five static TTF instances (Regular, Italic, SemiBold, Bold, BoldItalic) from `productiontype/Newsreader`, bundled in `crates/eidola-gui/assets/fonts/Newsreader16pt-*.ttf`, embedded via `include_bytes!`, and registered with `cx.text_system().add_fonts`. License at `assets/fonts/OFL.txt`.

We ship statics rather than the variable upright + italic because **gpui's macOS text system does not apply variable-font weight axes**: `gpui_macos::text_system::add_fonts` registers each TTF as one face with the properties of its default instance, and `font_kit::matching::find_best_match` picks the closest face per weight request. With only the variable TTFs registered, every weight request — `**strong**` (BOLD), headings (SEMIBOLD/BOLD), etc. — resolved to the Regular default and rendered un-bold. Five static faces make `find_best_match` pick correctly. Family name in the theme is `"Newsreader 16pt"` (the typographic family — nid 16 — that all five faces report; SemiBold sets nid 16 explicitly to override its nid 1 = `Newsreader 16pt SemiBold`, the canonical workaround for the Windows OS/2 4-style-per-family limit).

## Chat typography — book metaphor

The chat is the long-form reading view, so it gets typography of its own rather than inheriting the UI baseline. Settings/wallet/account stay at the 16px UI size; chat body is governed by three constants in `chat.rs` (`PROSE_FONT_SIZE`, `PROSE_LINE_HEIGHT`, `PROSE_MAX_WIDTH_REM`) plus a custom `markdown_style` for `TextView::markdown`.

- **Body 17px / line-height 1.65×.** Newsreader at 17 sits in proportion with a real book page; 1.65 is the readability sweet spot for serifs at this size. Both are applied on a `prose()` wrapper around every `TextView::markdown` call so the markdown renderer inherits them through the normal text-style cascade — gpui-component's `TextViewStyle` itself doesn't expose body size or leading.
- **Centered measure, ~640px wide** (`max_w(rems(40.))`, anchored to the theme's 16px rem so it stays absolute regardless of the prose font bump). Lands at ~65–72 chars/line, the canonical long-form measure. Centering uses a `prose_row()` h-flex with `justify_center` rather than `mx_auto`, because v_flex children stretch to full width by default.
- **Heading scale anchored to body, gentler ramp.** `heading_base_font_size = PROSE_FONT_SIZE` (instead of the gpui default 14px), and a callback returns h1 1.5× / h2 1.25× / h3 1.125× / h4–h6 1.0×. The default scale (h1=2×) reads like a marketing page; a book has a flatter type ramp where weight (BOLD/SEMIBOLD) carries most of the hierarchy.
- **Paragraph gap 1.5 rem** in the markdown renderer (vs gpui-component's default 1.0). About 85% of a body line — clear paragraph breaks without the run-on tightness of a chat tool.
- **Chapter delimiters between turns**, not alternating backgrounds. `chapter_delim()` renders a hairline `theme.border` rule across the prose column, broken in the middle by a small italic participant label ("You" / "Eidola" / "Error") in `text_sm` at `theme.muted_foreground`. Errors swap the label color for `theme.danger`. The whole chat sits on one uniform `theme.background` — speaker differentiation is carried by the rule and label, not by tinted bands. Rows themselves no longer carry vertical padding; the delim's `pt_8 + pb_6` is the entire inter-turn rhythm. The messages column's trailing `pb_8` keeps the last row off the input border.
  - **The first message has no leading delim** (`if idx > 0 { … }` in the message render loop). The user's text is the *start* of the page; only the first speaker change (the assistant's response) needs a label to introduce the new voice. Subsequent same-role turns (rare but possible after errors) still get their delim so the rhythm holds.
  - Layout: an outer `h_flex().w_full().justify_center()` centers an inner `h_flex().w_full().max_w(rems(40))` capped at the prose column. The inner flex has `flex_1` rule divs on either side of the label, sharing the leftover space.
  - **Scroll container invariant**: the `div().id("scroll").flex_1().overflow_y_scroll()` wrapping `messages_col` in `Render` *must* also have `.w_full()`. Without it, taffy content-sizes the scroll div instead of stretching it cross-axis — every descendant inherits that collapsed width, and the delim's `flex_1` rules disappear because there's no leftover space to grow into. The body's row stays full-width because its own content (code blocks) forces the scroll div to expand to that intrinsic width, but a delim with no wide content collapses. Diagnostic signal: in narrow windows the delim rule is visible only on the left side of the label, OR not at all — even though wider windows render fine because some other content has forced the scroll div wide enough. Snapshot regression coverage at 480 / 680 / 820 / 1400 px in `tests/visual/cases.rs`.
- **Inline code sizing.** Mono text at the same nominal size as a low-x-height serif reads visually larger, so inline code wants ~0.9× of body — but neither renderer can do per-span sizing. In the **composer** the fix is family-based: `composer_markdown_style` sets `inline_code_font_family("Courier New")`, whose x-height (0.423 em) matches Newsreader's (0.426 em) almost exactly, where Menlo (0.547 em) read ~28% larger; fenced code blocks keep Menlo. In the **transcript**, gpui-component's `TextView` renders inline code as body-font text with a `theme.accent` chip (its `style.code` mark only sets `HighlightStyle::background_color` — gpui's `HighlightStyle` has no font family or size fields), so it's size-consistent with body by construction but *not* mono — a known TextView gap with no knob to close it. The editor's chip default is also `theme.accent` so composer and transcript chips match.

Knobs we don't touch yet / TextView gaps: per-span font size in either renderer (gpui's `TextRun`/`HighlightStyle` carry no size — upstream limitation), mono inline code in the transcript (see above), heading top breath specifically (h2's `pb(rems(0.3))` is hardcoded; headings inherit the previous block's `paragraph_gap` as their top space), TextView code-block padding beyond the `code_block` StyleRefinement, and OpenType figure features (`onum` for oldstyle figures would be book-y for prose but conflicts with technical content like version numbers). List-item spacing is *not* a gap anymore: the transcript's `TextView` already stacks items flush (no inter-item gap), and the composer tightens items to 0.35× of the paragraph gap via `MarkdownStyle::list_item_gap_factor`.

**Persistent participant indicator (designed with the delim).** The chapter delimiter identifies the speaker *at the boundary*; once a long assistant response is partway scrolled past it, the delim is off-screen and the reader loses the cue. The persistent participant indicator carries that identity *forward*: a small italic delim-voice label ("You" / "Eidola" / "Error", `text_sm` muted, error turns in `theme.danger`) pinned to the **left** side of the title-bar band (the ⌥-revealed model label owns the right; both are quiet absolute chrome, so neither reflows the page). The two are one system — the delim owns the in-page cue, the indicator the out-of-page one — so they never compete: the indicator is visible **exactly when** the page-local cue is missing and hidden the moment it returns.

The visibility derivation (`ChatView::derive_participant_indicator`, a pure function of `list()` scroll state + the item model, behavior-tested via `derive_participant_indicator_at`) keys off the item at the logical scroll top: hidden at the page top (`item_ix 0`, offset 0); hidden while the scroll offset into the top item is still within the leading-delim band (`DELIM_BAND_HEIGHT`, the delim's `pt_8 + label + pb_6` height — the real delim is on screen); otherwise it surfaces that item's voice (also for a delim-less first message, whose opening line scrolling up is itself the "cue is gone" signal). Over the composer item it returns `None` — the writing zone is the reader's own frame, not a turn to label. It is recomputed each frame in `render` from the post-paint scroll position and stored on the view so the band reads one consistent value.

## Inline composer — WYSIWYG markdown editor

The chat has no separate input bar. The composer is a `MarkdownEditor` entity from `crates/gpui-markdown-editor/`, dropped in as the last child of `messages_col` inside the scroll container. It shares the prose column, prose typography (Newsreader 17px / 1.65× / 1.5 rem paragraph gap / gentle heading scale — see [`composer_markdown_style`](src/chat.rs)), and the same `theme.background` as the rest of the page. What the user types is rendered with full markdown styling — bold, italic, headings, lists, fenced code, links, math, … — and shapes pixel-for-pixel like the assistant's reply will when it lands in the transcript.

A "You" `chapter_delim` is rendered above the composer **only when there is preceding content** (`!messages.is_empty() || streaming.is_some() || error.is_some()`). On a fresh, empty page the cursor sits at the top with no header — like opening a blank notebook.

There is no Send button. ⌘↩ is the sole submit path.

**Indefinite growth, single scroll surface.** The editor renders one gpui block per markdown block in a vertical flex column; there's no internal scrollbar. The editor grows naturally with content. The composer is the **final item of the virtualized transcript `list()`** (see [Transcript virtualization](#transcript-virtualization--the-chat-on-list)) — so messages + composer remain one continuous scroll surface, but the scroll is now owned by the `list()` element (`ListState`), not a wrapping `overflow_y_scroll` div. The composer item is registered with the editor's focus handle via `splice_focusable`, so when it scrolls offscreen while focused the `list()` keeps rendering it for keyboard interaction — the book-page feel survives virtualization.

**Empty-state floor.** With markdown `""`, the editor's render pipeline emits no blocks and its container collapses to zero height — which would make the composer un-clickable after losing focus. `prose().min_h(PROSE_FONT_SIZE * PROSE_LINE_HEIGHT)` puts a one-body-line floor on the editor's wrapper so the empty state stays clickable. Once any character is typed, normal content height takes over.

**Half-viewport bottom padding.** The composer's wrapper carries `pb(window.viewport_size().height * 0.5)` (computed at render time). The cursor never sits pinned to the bottom edge — there's always a half-page of empty space below the active line, so the parent scroll can keep the typing zone in the comfortable middle of the viewport as content grows. `messages_col` no longer carries `pb_8`; the wrapper's pb is the bottom breath. Computed in `Render` rather than as a constant because it tracks live window resizes.

**Cmd+Return dispatch.** ⌘↩ resolves to `Self::submit` via gpui's normal action dispatch. The editor's `MarkdownEditor` key context (see `install_markdown_editor_keybindings` in `lib.rs`) **does not bind `cmd-enter`** — only plain `enter` (newline insertion) and `shift-enter` (line break). So the ChatView-context `cmd-enter → Send` binding remains the innermost matching entry in the focus chain whenever the editor has focus, and `Self::submit` (registered with `cx.listener(Self::submit)` on the v_flex root) fires.

`Self::submit` is idempotent under streaming (`if self.streaming.is_some() { return; }`). The composer remains visible and editable during streaming, but submit is silently a no-op until the stream completes — known UX hole, fine for first iteration. `submit` reads `editor.state.markdown.trim()`, sends the raw markdown source upstream, and resets `editor.state = EditorState::default()` to clear the composer.

**Focus on construction.** `ChatView::new` calls `window.focus(&editor_focus, cx)` against the editor's own focus handle so the cursor lands in the composer when a window opens. The view's own `focus_handle` is still `track_focus`ed by the root v_flex (it's what behavior tests dispatch through), but no longer focused at construction — the editor takes that role.

**Key context routing.** `install_markdown_editor_keybindings` in `lib.rs` binds the editor's actions (Backspace, Delete, Enter, Tab/Shift-Tab, arrow keys, word- and line-granular motion and deletion, ⌘A/X/C/V/⌘⇧V) under `Some("MarkdownEditor")`. Settings/account still use `gpui_component::Input`, which establishes its own `Input` context with parallel bindings, so the two surfaces coexist without conflict. The Edit menu routes Cut/Copy/Paste/Select All as `gpui_component::input::*` action types via OS selectors; `MarkdownEditor::render` now also registers `on_action` handlers for these same `gpui_component::input::*` types (forwarding to its own method implementations), so the Edit menu works when the composer has focus — the formerly known gap is closed.

**Test wrapping invariant.** Behavior tests still wrap the view in `gpui_component::Root` (see `tests/behavior.rs::open_view`) even though the chat composer no longer uses `gpui_component::Input` — keeping Root mirrors production (`lib.rs::open_main_window`) and is required for any test that exercises SettingsView. The chat tests no longer *require* Root strictly, but the helper is shared across views so removing the wrap would split the harness.

## WindowInput — per-window modifier state

`src/window_input.rs` — `WindowInput` is a **per-window gpui entity** that holds the live keyboard modifier state for one window. It exists because `ModifiersChangedEvent` in gpui (at our pinned commit 969a67fc) dispatches along the **focused element's ancestor path only** — a listener on a sibling branch of the focused node never fires. Without `WindowInput`, a Settings pane's `on_modifiers_changed` listener was dead while a text input in a different pane had focus (wave-2 bug 2).

**Pattern — one listener per window, registered on the root view only:**

- Each `open_*_window` builder creates one `WindowInput::new(cx)` and passes it to the view constructor.
- The root view (`ChatView::render` root v_flex, `SettingsView::render` root h_flex) registers the window's **single** `on_modifiers_changed` listener and calls `wi.update(cx, |wi, cx| wi.update_modifiers(event, cx))`.
- Descendant views (`GeneralView`) observe the entity (`cx.observe(&window_input, …)`) and read `window_input.read(cx).alt_held()`. They never register their own `on_modifiers_changed`.

**Rule — enforced structurally, not by convention alone:** No view below the root may call `.on_modifiers_changed(…)` anywhere in its `Render` impl. The `WindowInput` entity is the only mechanism for sharing modifier state within a window. Adding a new ⌥-affordance in any descendant view means: observe `window_input`, never add a second listener.

**Staleness:** gpui at 969a67fc has no window-activation or "window became/resigned key" observer hook. If the user switches away while holding ⌥, `alt` may linger `true`. This is self-healing: the next `ModifiersChangedEvent` in this window writes the current state. The stale `true` is transient and harmless (the ⌥ reveal is a cosmetic affordance with no irreversible action).

**Test helper:** `WindowInput::set_alt_for_test` mutates `modifiers.alt` directly (no `ModifiersChangedEvent` construction) for snapshot tests that need to render the revealed state without synthesizing platform events. Behavior tests use `VisualTestContext::simulate_modifiers_change` to drive the real dispatch path through the root listener.

## Model picker — Option-revealed

The first real exercise of the "Option key reveals power" pillar; the interaction pattern here is the template for future ⌥-affordances. At rest the page shows **no model chrome at all** — the blank page stays sacred.

- **Reveal.** The chat window's `WindowInput` entity (see [WindowInput — per-window modifier state](#windowinput--per-window-modifier-state)) carries the live ⌥ state. `ChatView` holds `Entity<WindowInput>`, observes it for re-render, and reads `window_input.read(cx).alt_held()` inside `model_revealed(cx: &App)`. The single `on_modifiers_changed` listener lives on the root v_flex in `ChatView::render`; it calls `wi.update(cx, |wi, cx| wi.update_modifiers(event, cx))`. gpui dispatches `ModifiersChangedEvent` along the focused element's ancestor path; the root is always an ancestor of the always-focused composer, so the listener fires for every modifier transition while the window is key. While ⌥ is held (or the picker is open — the open panel keeps its anchor), the current model id appears right-aligned **in the title-bar band** (`render_title_bar`), `text_sm` muted italic — the same voice as the chapter-delim labels. The band is absolutely-positioned chrome painted last, so the reveal cannot shift page layout. Model chrome renders only on the `Ready` onboarding stage; the welcome/plans pages have no sends for it to describe.
- **Picker.** Clicking the revealed label or pressing **⌥⌘M** (`chat::ToggleModelPicker`, bound in the `ChatView` key context) toggles a quiet panel anchored under the band's right edge, listing `ModelsStore` with honest per-model info straight from the `/models` payload: context length plus the credit rates actually charged (`ModelInfo::{prompt,completion}_credits_per_token`, or the flat `request_credits` for per-request models; if pricing is absent we show context length only — no invented numbers). The current selection and the config default are marked; rows are hairline-separated; `popover_style` keeps the surface consistent with gpui-component overlays. Click-out dismisses (`on_mouse_down_out`), ⌥⌘M toggles closed. Hand-rolled rather than `gpui_component::Popover` because the open state must live on the view (⌥⌘M and behavior tests drive it without a pointer), the anchor is the absolute band rather than an in-flow `Selectable` trigger, and the band already paints above the scroll content, so plain absolute positioning does everything `deferred(anchored(…))` would.
- **Scope of the choice.** `Space::selected_model: Option<String>` is **per-space** (it lives on the shared entity, so both windows on one space see the same selection); `None` means "follow the config default". `ChatView::current_model(cx)` resolves the space's selection → `ConfigState::default_model` → embedded fallback (`eidola_app_core::config::DEFAULT_MODEL`, for stub stores). The default lives in config as the `default_model` override (`Config::default_model()` resolver, `AppCore::set_default_model`, surfaced through `ConfigState`); the picker's footer shows a small secondary "Set … as default" affordance only when it would change anything. Switching mid-stream applies to the *next* send — the in-flight request is never hot-swapped (the runner captured its model at submit).
- **Honesty plumbing.** `ChatView::submit` resolves the model once and calls `Space::submit`, which records it in `last_submitted_model` *before* its own stub-core guard (so behavior tests assert exactly what a real send would use) and passes it through `chat_stream` — also what app-core persists on the action row in the local DB.
- **Tests.** Behavior: `alt_reveals_model_label` (drives the real modifiers dispatch via `VisualTestContext::simulate_modifiers_change`), picker toggle round-trip, Esc/Up/Down/Enter keyboard navigation (`escape_dismisses_picker`, `picker_up_down_moves_highlight`, `picker_enter_selects_highlighted_model`, `picker_enter_with_no_highlight_is_noop`), default-vs-selection submit resolution, mid-stream selection. Config persistence round-trips are covered in app-core (`set_default_model_round_trips_through_config_state` + `config.rs` TOML tests). Snapshots: `chat_model_reveal`, `chat_model_picker`.
- **Deferred:** opacity fade-in on reveal (instant today; an animation would also make snapshots nondeterministic).

## Streaming chat

The **`Space` entity** drives chat via `bridge::chat_stream` (which wraps `AppCore::chat_stream` from `crates/eidola-app-core/`) inside its single `submit_runner` slot — not a detached view task. The core's streaming method posts `stream: true` to the OpenAI-compatible upstream and forwards each SSE chunk's `delta.content` as `ChatStreamEvent::ContentDelta` and `delta.reasoning_content` / `delta.reasoning` (vLLM-style) as `ChatStreamEvent::ReasoningDelta`. The terminal `ChatResult` is the function's return value, not an event — the channel just closes.

While streaming, `Space::streaming: Option<StreamingResponse>` holds the live `reasoning` + `content` buffers and a disclosure-`expanded` flag, and each delta emits `SpaceEvent::StreamDelta`. `ChatView` renders it (reading through `self.space.read(cx)`) as a single row below the user message: a clickable "Thinking…" header (a `Button` styled with a chevron) whose toggle calls back into `Space::toggle_streaming_reasoning`, the reasoning body when expanded, and the partial markdown content as it grows. On done, the `Space` drops `streaming`, reloads the transcript from the DB, re-attaches the captured reasoning to the new last assistant entry, and emits `StreamEnded` — only the final content is persisted; reasoning is **ephemeral by design** (not written to the local DB), and `merge_messages_from_db` preserves it by position across reloads. Because the streaming state lives on the shared entity, a second window on the same space watches the same buffers fill (wave-2 bug 4).

The CLI uses streaming too: `crates/eidola-cli/src/main.rs` pumps `ContentDelta` to stdout and `ReasoningDelta` to stderr (dimmed and prefixed with `thinking: ` when stderr is a TTY) so a piped stdout still captures only the final answer.

Refund handling for streaming differs from blocking only in *where* the refund token comes from: SSE responses have no inline JSON body to carry it, so the streaming path always goes through the `/v1/credentials/refund` recovery endpoint after the stream ends. Same recovery endpoint as the existing network-error fallback in the blocking `chat()`.

## Tail-on-bottom scroll

Tail-scroll is **window-local** (each window has its own `ListState` scroll position) and is now implemented on the `list()`'s `FollowMode::Tail` (the pinned zed rev's built-in follow-tail), not the old "rebuild-the-scroll-surface-each-frame" placeholder — the prior `on_next_frame`/no-persistent-`ScrollHandle` note is obsolete.

- **Engagement.** Follow is engaged on **submit** (`ChatView::submit` calls `set_follow_mode(Tail)` + `scroll_to_end`), not at construction — a reopened space must rest at its top (the pre-virtualization resting state), and `set_follow_mode(Tail)` immediately jumps the scroll to the end. So a fresh window with history rests at the top; the moment the user sends, the list follows the new exchange.
- **Streaming.** `ChatView::on_space_event` still matches `SpaceEvent::StreamDelta` as the per-chunk hook (distinct from the catch-all `MessagesChanged`/`StreamEnded` re-render — the doctrine's "tail-scroll only on `StreamDelta`"). On a delta it `remeasure_items` the streaming row (its index is stable) and, **only while `is_following_tail()`**, `scroll_to_end` — so the freshest token stays visible while a reader who has scrolled up is left undisturbed (`FollowMode::Tail` disengages the instant they scroll up, and re-engages when they scroll back to the bottom). `MessagesChanged`/`StreamEnded` reconcile the item model (`rebuild_transcript`) and re-pin only if still following.
- **24px at-bottom tolerance** is the list's own: `FollowState` treats `scroll_offset + viewport ≥ total_height − 1px` as "at bottom" and the wheel-scroll path disengages on any upward delta, so the policy ("at-bottom re-pins on deltas; scrolled-up is left alone; submit always brings the new exchange into view") falls out of the follow-mode machinery rather than hand-rolled offset math.

## Window model

**Chat windows are non-singleton; the space *entity* they share is not.** Every `NewSpace` opens a fresh `ChatView` via `open_main_window` → `open_chat_window(cx, stores, None)`; with `space_id: None` `ChatView::new` calls `SpacesStore::blank()` for an id-less `Space` (instant ⌘N; the registry adopts it once its first exchange assigns an id). The Library reopens via `open_space_window(cx, stores, space_id)` (`Some(id)`): `ChatView::new` calls `SpacesStore::open(id)`, which **joins** the registry — so opening the same space twice yields two independent *windows* over **one shared `Space`** (transcript, streaming, submit, model selection are common; submits/streams in one window appear in the other). The composer draft stays per-window by design. There is still no window *dedup* (two windows is fine, acceptable v1) — the change from before is that they no longer each hold a private transcript copy. `open_chat_window` calls `cx.activate(true)` after `cx.open_window` so a window opened from another app's context (dock right-click while a different app is foreground) brings Eidola to the front rather than opening behind.

**Settings, Library, Updates, Record, and About are singletons.** `AppGlobal.settings_window` / `AppGlobal.library_window` / `AppGlobal.updates_window` / `AppGlobal.record_window` / `AppGlobal.about_window` (`Option<WindowHandle<Root>>`) cache the handles, and `OpenSettings` / `OpenLibrary` / `CheckForUpdates` / `OpenRecord` / `About` raise the existing window via `window.activate_window()` if it's still open (shared helper `try_focus_existing_singleton`, parameterized over the `AppGlobal` slot). Both open paths are **synchronous** (via `App::open_window`) so the cache is populated before the handler returns. Liveness is checked by matching the cached `WindowId` against `cx.windows()` (the authoritative live list) — borrowing Zed's pattern, except Zed can use `AnyWindowHandle::downcast::<SettingsWindow>` directly because their settings root is uniquely typed; ours is `gpui_component::Root` (shared with chat windows), so we match by id instead. A stale id self-heals on the next invocation — no `on_release` bookkeeping needed. `OpenLibrary` additionally calls `SpacesStore::refresh` on every invocation (not just first open) so a long-lived Library window doesn't show a stale listing. `CheckForUpdates` follows the same singleton pattern via `AppGlobal.updates_window`, and re-runs `UpdateStore::check_now` when raising an existing window so the menu gesture always produces a live check.

## Updates window

`updates.rs` renders the verified update-notification flow (`eidola_app_core::updates` — see the workspace `AGENTS.md` for the verification model and failure-mode matrix). Small singleton (480×360 default), opened via Eidola menu → "Check for Updates…" (directly under "About Eidola", the standard macOS placement).

- **State is derived, not stored**: `UpdatesView::derive_display(update_store)` maps the `UpdateStore` snapshot + its `checking` flag to a `UpdatesDisplay` enum — `Checking` (only when a check is genuinely in flight *and* there's nothing else to show; a re-check over a standing result keeps the result up and just disables the footer button into "Checking…"), `NoneYet`, `UpToDate` (with last-checked relative time), `UpdateAvailable` (one primary action: "View Release…" → `cx.open_url`), `Unverifiable` (danger-tinted security band quoting the exact failure reason; deliberately **no** link to the artifact), `ClaimsChanged` (side-by-side expected/attested delta table; the only affordance is the explicit, non-primary "Treat as Update" → `UpdateStore::accept_changed_claims`), and `CheckFailed` (quiet muted copy — an offline blip is not a security signal).
- **Background polling**: `run()` calls `UpdateStore::start_polling` at launch (app-core runs one check immediately + every ~6h on its tokio runtime, persisting results). The view constructor calls `UpdateStore::load_last` so a poll result that landed while no window was open is reflected on next open — no banners in chat windows.
- **Store plumbing**: `UpdateStore` holds the cached `snapshot` + `checking` flag; `check_now` owns its own supersede task slot and its `checking` flag makes re-entry a no-op while a check is in flight (so the constructor-triggered check and the menu handler's re-check are naturally idempotent). There is no shared busy gate.
- Behavior tests cover the display derivation for every matrix row plus stub no-op actions; visual snapshots cover all six states (`updates_*`), day+night.

## Library — table of contents, not a sidebar

`library.rs` renders the space listing as a book's table of contents (520×620 default): one column, hairline `theme.border` rules *between* entries (border-top on `idx > 0`, no boxes/cards/avatars), each row a title — or, for untitled spaces, the first-message snippet in `theme.muted_foreground` — with a right-aligned relative date ("today" / "3d ago" / "2w ago"; see `relative_date`) in `text_sm` muted. A chapter-delim-style heading ("Library", italic, between hairline rules) echoes the chat's chapter delimiters so the window reads as another page of the same book. Empty state is a single centered muted line. The data comes from `SpacesStore` (the `Loadable<Vec<SpaceInfo>>` index, archived excluded), refreshed via `SpacesStore::refresh`.

**Virtualized.** The listing renders through gpui's `uniform_list` (see [Virtualized lists](#virtualized-lists)): rows are fixed-height (`ROW_H`, single line), the closure (`render_rows`) is a dumb indexer that clones only the visible slice from `SpacesStore`, and the list self-scrolls via a stored `UniformListScrollHandle`. The centering wrapper is `h_flex().flex_1().min_h_0().justify_center()` around the list, which carries `.h_full().w_full().max_w(rems(34))` so it fills the height (so it actually scrolls) while staying capped at the prose measure and centered — a library too long for one screen now renders O(visible), not O(rows).

Interactions: clicking a row opens the space (`open_space` defers `open_space_window` via `cx.defer` so the new window opens after the current update cycle); hovering a row (`hovered: Option<usize>` tracked via `on_hover`) reveals a ghost × button in a fixed-width slot (so its appearance doesn't shift the date column) that archives via `SpacesStore::archive` — which removes the cached row *immediately* (so the UI responds without a backend round-trip, and so stub-store tests exercise the local path) and then runs the real archive + re-fetch. The archive click calls `cx.stop_propagation()` so it doesn't also open the row. `archive` / `open_space` are public methods so behavior tests drive them directly instead of synthesizing mouse events; `set_hovered_for_test` exists for the hover snapshot.

**Inline rename.** Double-clicking a row calls `LibraryView::begin_rename(space_id, current_title, window, cx)`, which creates an `Entity<InputState>` seeded with the current title, focuses it immediately, and subscribes to `InputEvent`: `PressEnter` → `commit_rename` (writes through `SpacesStore::rename`, which optimistically updates the cached row and then calls `AppCore::rename_space` + re-fetches); `Blur` → `cancel_rename` (drops the draft without writing). Pressing Escape inside the `InputState` propagates through (the `Input` context doesn't capture Escape by default when `clean_on_escape` is false), which causes the input to lose focus and fires `Blur`, which cancels. The active rename space is tracked in `renaming: Option<(String, Entity<InputState>, Vec<Subscription>)>` on the view; `renaming_space_id()` returns the id for tests. The archive slot is hidden while a row is being renamed so the × doesn't compete with the input. `SpacesStore::rename` uses the same optimistic-update + supersede-slot pattern as `archive`; the bus-driven `Change::SpaceIndex` reconciles all windows.

## Settings — calm two-pane, ⌥-revealed advanced

`SettingsView` (620×520) is an `h_flex`: a narrow nav band (`NAV_WIDTH` 132px) on `theme.sidebar` with quiet text items (active = `sidebar_accent` pill), and the selected pane in the content column. Settings keeps **no raw-data dumps** — summaries link to the Record window.

- **Scroll-width invariant (again).** The content pane's scroll container is a `v_flex().flex_1().h_full().min_w_0()` wrapping `div().id("settings-body").w_full().flex_1().overflow_y_scroll()`. Without the `w_full` on the scroll div, taffy content-sizes the pane: long lines refuse to wrap and rows shrink to content width (same failure mode as the chat transcript invariant).
- **General**: the Base URL row shows the resolved value plus an honest source line — "Built-in pin" vs "Override — the built-in pin is …" with a one-click "Revert to pin" (`AppCore::clear_base_url_override`; `ConfigState` carries `base_url_pin` / `base_url_is_override`). Saving a value equal to the pin is treated as a revert. The **advanced rows** (attestation URL, domain separator, hardware CAs, trusted-measurements summary + Record link) render only while **⌥ is held**: `SettingsView`'s root `h_flex` registers the window's single `on_modifiers_changed` listener and mirrors events into `Entity<WindowInput>` (see [WindowInput — per-window modifier state](#windowinput--per-window-modifier-state)); `GeneralView` observes `WindowInput` and calls `set_advanced` when `alt_held` changes. `GeneralView` **never** registers its own `on_modifiers_changed` — doing so would be dead while a text input in a sibling pane (Account / Wallet) or in General's own Base URL field has focus, because `ModifiersChangedEvent` dispatches along the focused element's ancestor path only (wave-2 bug 2). A one-line italic hint discloses the affordance at rest. The measurements row never shows hex — it links to the Record via `window.dispatch_action(Box::new(OpenRecord), cx)`.
- **Account**: balance + pools (humanized future expiries, `humanize_expiry`), plans via the shared `plans::plan_rows` with the same checkout flow as onboarding (`AccountStore::request_checkout`, view-owned pending state), and Reset behind a **two-step inline confirm** (`request_reset` → armed warning + "Reset account" / "Keep account"; no modal). The manual allocate-credits input was removed — app-core auto-provisions credentials from balance, so it no longer justified its row. **Account create/reset failures are surfaced, not dropped** (honest-states rule): `AccountStore::create_account` / `reset_account` store the `Err` in `account_op_error` (cleared at the start of the next attempt and on success), and `AccountView` renders it in the `error_banner` style directly under the Account controls so the button never silently does nothing. Hidden by default, so the `settings_account` snapshots are unchanged at rest. Behavior test: `account_op_error_surfaces_and_clears`.
- **Wallet**: one history listing from `WalletStore` (via `WalletStore::refresh` → `AppCore::wallet_lifecycle`, the local `credential_lifecycle` view): active / in flight (warning color, recoverable) / spent / expired (muted). "Recover in-flight" appears only when something is actually in flight.

## The Record window — the trust door's engine room

`record.rs` (`RecordView`, singleton, ⇧⌘L / File → "Record…", 860×640) exposes the raw local Turso trail. Three sections in a quiet text strip that doubles as the title bar (`STRIP_LEFT_PAD` 80px clears the traffic lights; right side carries a Refresh affordance and an italic "The Record" wordmark):

- **Attestations** — `attestation` rows (middle-truncated hash, pcr digest, connection count, doc size, UTC timestamp). Click → the full raw document: pretty-printed JSON when it parses, UTF-8 text, hex otherwise.
- **Requests** — `request` ⋈ `connection` rows (mono `METHOD /path`, status colored by outcome, transport/duration/attempt/credential/error subline). Click → full detail: kv rows (the attestation hash links through to its attestation detail) + raw request/response headers and bodies.
- **Spending** — the `spend_trail` view grouped by credential (consecutive same-nonce rows share a group header with held/charged amounts; rows are time-ordered so the grouping matches reality). Rows click through to the request detail.

Mechanics worth knowing:

- **Windowed fetch.** `Listing<T>` per section; each page asks app-core for `PAGE+1 = 51` rows (`LIMIT/OFFSET`) to learn `has_more` without a COUNT. Data flows through the `bridge` free functions (`bridge::list_attestations` / `list_requests` / `spend_trail` / `attestation_detail` / `request_detail`, taking the `Arc<AppCore>` from `Stores::app_core()`) so the view owns its loading state in its own task slots. These window-scoped reads become a reader entity in a later step. Stub stores skip the fetch; tests install rows via `set_*_for_test`.
- **Virtualized listings + flat display model.** Each populated section renders through `uniform_list` (see [Virtualized lists](#virtualized-lists)), so frame work is O(visible), not O(loaded) — the wave-2 bug-3 fix (the window used to slow to a crawl as "Load more…" appended pages, because every row re-rendered every frame). To keep the `uniform_list` closure a dumb indexer, each `Listing<T>` precomputes a flat `Vec<DisplayRow>` (`Header(i)` / `Data(i)` / `LoadMore`) on every state change (`rebuild_flat_display` for attestations/requests; `rebuild_spending_display` for the grouped Spending view). **Spending group headers survive virtualization** because they are precomputed `Header` rows interleaved into the display model, not a render-time `prev_nonce` branch the windowed closure couldn't reconstruct. All rows — data rows, group headers, and the trailing load-more row — are normalized to one fixed height (`ROW_H`); the formerly taller spending group headers are slightly tighter now (even spacing carries the group rhythm instead of extra top padding — an intentional, documented change). **Load-more stays windowed:** the trailing `DisplayRow::LoadMore` is an explicit, quiet "Load more…" row that appends the next 51-row page; while a fetch is in flight it renders an honest "Loading more…" in-flight row (it maps to a real task, never an infinite spinner). The empty/loading/detail bodies stay in an ordinary `overflow_y_scroll` container (`scroll_wrap`); only a populated listing is the self-scrolling `uniform_list`.
- **Raw payloads are never redacted** (the user's own traffic on their own machine). Inline rendering is capped at 64 KiB with an honest note; the per-section **Copy** affordance always writes the *full* payload to the clipboard. Raw text renders selectable through `TextView::markdown` inside a code fence whose backtick run is computed to exceed any run in the content (`fenced`); the `TextViewStyle` mirrors `theme.mode.is_dark()` so the block ground tracks Circadian night.
- **Parse caching (no per-frame projection).** Opening a detail builds a `DetailCache` **once**: the raw-payload projection (pretty-printed JSON / UTF-8 / hex dump), the `fenced` + 64-KiB-capped inline string, and the full Copy string are all computed at open time and stored as `CachedPayload`s. The render path (`raw_section`) only reads the cache — it never re-projects per frame. This is the other half of bug-3: a large attestation document or response body used to re-run `serde_json::to_string_pretty` / `hex_dump` / `fenced` on every frame. **TextView itself does not re-parse per frame at our pin** (verified in `gpui-component@dadfca9`): `TextView::markdown(id, text)` resolves its `TextViewState` via `window.use_keyed_state` keyed on the element `id`, and `TextViewState::set_text` early-returns when the text is unchanged — so a *stable id + stable string* parses exactly once. The per-frame cost was always our own Rust-side projection, which the cache removes; passing the cached `fenced` string (identical bytes each frame) under the payload's stable `id` keeps TextView's parse a one-time cost. Perf is guarded by `record_frame_work_is_constant_in_loaded_rows` (behavior test: 1 page vs 10 pages — the visible-window render produces the same fixed row count and does not scale in time).
- **Space cross-link.** The request detail view includes a "Space" kv row when `RequestDetail.space_id` is set (a LEFT JOIN through `action → space` added to `db::get_request`). The row renders as a quiet link (`theme.link` color, hover underline) whose click defers `open_space_window(cx, stores, space_id)` — the same path the Library uses, so opening a linked space from the Record is a single click. The label is `space_title` when present, the bare id otherwise. Behavior test: `record_request_detail_exposes_space_link`.
- **Mono is for data, the UI font is for chrome** — `mono()` (Menlo 12/13px) wraps every hash, nonce, path, and payload; labels, timestamps and empty states stay in Newsreader. Timestamps are UTC and say so (`fmt_utc`, Hinnant `civil_from_days` — no chrono dependency).

## Virtualized lists

Fixed-height lists render through gpui's `uniform_list` (`docs/architecture/state.md` → "Lists"): the **Record** sections (attestations / requests / spending) and the **Library** index. The **chat transcript** is variable-height and renders through `list()` instead (see [Transcript virtualization](#transcript-virtualization--the-chat-on-list)). The `uniform_list` idiom (mirrors zed's project panel / pickers at our pin):

- **Item-range closure via `cx.processor`.** `uniform_list(id, count, cx.processor(|this, range, _window, cx| this.render_rows(range, cx)))` — the closure is a *dumb indexer*: it renders only `range` (the visible window), so per-frame work is O(visible), never O(loaded). Any structure the closure would otherwise have to reconstruct from a bare range (group headers, separators, a load-more affordance) is **precomputed into a flat display model** (`Vec<DisplayRow>` in the Record) so the closure stays a pure index→element map.
- **Fixed row height.** `uniform_list` measures one row and lays every row out at that height, so all rows in one list must be the same height. Rows are normalized to an explicit `.h(ROW_H)` (deterministic, not font-metric-coupled); variable content (multi-line sublines, depth) is truncated with the middle-ellipsis helpers and pushed into the detail view — the listing row is always fixed-height.
- **Stored scroll handle.** A `UniformListScrollHandle` lives on the view and is passed to `.track_scroll(&handle)` each frame (so scroll position survives re-renders and supports programmatic `scroll_to_item`).
- **Sizing.** `ListSizingBehavior::Auto` (the default) makes the list fill its flex parent and scroll internally — place it as a `flex_1` child (v_flex parent) or give it `.h_full()` inside a `flex_1().min_h_0()` parent (h_flex, e.g. the Library's centering wrapper). The list's own `overflow.y = Scroll` is set by `uniform_list`; do **not** wrap it in another `overflow_y_scroll` div (that nests two scroll surfaces and the inner list collapses).

## Transcript virtualization — the chat on `list()`

The chat transcript has *variable*-height rows (a one-line user turn vs a long markdown answer with code blocks), so it uses gpui's `list()` (`ListState`, measured items) rather than `uniform_list`. This is the variable-height analogue of the Record/Library pattern above — and the structural fix for the transcript rendering every message every frame.

- **Flat item model.** `ChatView::transcript_items: Vec<TranscriptItem>` is the chat's flat display model (the `Vec<DisplayRow>` analogue): one entry per `list()` item — `Message { index, leading_delim }`, `Streaming`, `Error`, `Composer { leading_delim }`, in render order. The `list()` render closure is a dumb indexer: it upgrades a captured `WeakEntity<ChatView>`, `update`s back in, and dispatches `render_transcript_item(ix)` to the per-item renderers (`render_message_item` / `render_streaming_item` / `render_error_item` / `render_composer_item`). Per-frame work is O(visible). The perf guard is `transcript_render_work_is_constant_in_message_count` (20 vs 500 messages → same fixed 12-item visible window, flat per-frame time).
- **The chapter delim travels *with* its item.** The delimiter that introduces a turn is rendered *inside* the same item as the body it labels (not as its own item), so the two measure and scroll as one unit and a delim never lands alone on an item where it could collapse. The first row has no leading delim (`leading_delim: false`), exactly as before.
- **Width-collapse invariant is *stronger* under `list()`.** Each item lays out at a definite full width (`list()` passes `AvailableSpace::Definite(width)` per item), so the delim's `w_full → prose_row → flex_1` rules stretch correctly per item — there is no single `overflow_y_scroll` container whose missing `w_full` could content-size the whole tree. The 480 / 680 / 820 / 1400px snapshot cases re-verify this.
- **`rebuild_transcript(pin_tail)`** reconciles `ListState` against a freshly-computed model, **correctness-first**. The tail-append fast-path (`splice_focusable` only the new suffix, preserving the reader's exact scroll position) fires **only** when the *entire* old model — its trailing suffix included — is an unchanged prefix of the new one (items added strictly after it) **and** the list is already in sync with the old model. Every other transition `reset`s + re-splices wholesale so the list's item count and per-item measurements can never drift from the model. This is deliberately stricter than the original "prefix match" heuristic, which was both a dead fast-path (in this UI the trailing composer shifts on *every* real turn, so a turn append is never a pure prefix-extension) and a latent footgun: the first submit reshape `[Composer] → [Message, Streaming, Composer]` *looks* like a tail-append (composer is last; new items appear before it) and a prefix-blind splice would leave `ListState` desynced — which is exactly the wave-4 regression where the first user message vanished from the transcript the instant streaming began (persisted fine; the list just stopped measuring/painting it). Because `is_tail_append` also requires `in_sync`, a list that ever desyncs self-heals on the next reconcile (it forces a `reset`). The composer suffix item is spliced with the editor's focus handle (offscreen-focused composer keeps rendering). It runs from the `observe(&space)` (model-only reconcile, `pin_tail=false`), the `StreamEnded`/`MessagesChanged`/failure hooks (`pin_tail=true`), and the constructor seed. Regression replays: `first_message_survives_streaming_start` (blank window first submit + streaming start + stream-end reload) and `second_submit_in_existing_space_keeps_all_messages` (the splice-vs-composer interaction in an existing space); the O(visible) perf guard `transcript_render_work_is_constant_in_message_count` stays green.
- **Top padding, not a wrapping scroll div.** The traffic-light reserve is the `list()`'s own `.pt(TITLE_BAR_RESERVE)` (accounted in its layout), and the list is `.size_full()` inside a `flex_1().min_h_0()` div with **no** `overflow_y_scroll` — the `list()` owns scrolling. The title-bar band still paints absolutely over it (the page root is `relative`).

## Edge-to-edge titlebar

`transparent_titlebar()` returns `TitlebarOptions { appears_transparent: true, title: None, traffic_light_position: Some(point(14, 11)) }`. macOS extends the content view under the traffic-light buttons and stops painting a separate titlebar background. Each view leaves room at the top so the lights don't land on real UI:

- **`chat::TITLE_BAR_RESERVE`** (36px on macOS): vertical reserve, plus a `theme.background → transparent` linear-gradient overlay (`ChatView::render_title_bar`) painted absolutely over the scroll area. Messages scrolling up under the band fade smoothly into the chrome instead of clipping at a hard edge. The band hosts two quiet, coexisting cues — the **left** carries the persistent participant indicator (see Chat typography), the **right** the ⌥-revealed model label and anchored picker panel (see [Model picker](#model-picker--option-revealed)). Both are absolutely positioned chrome, so neither reflows the page or shifts the other.
- **`settings::NAV_TOP_RESERVE`** (44px on macOS): the traffic lights sit over the sidebar nav band, so the band's first item starts below them.
- **`record::STRIP_LEFT_PAD`** (80px on macOS): horizontal pad on the Record's section strip, which doubles as the title bar — the lights live to its left. 80px matches gpui-component's own `TITLE_BAR_LEFT_PADDING`.

## macOS UX — menus, keybindings, action dispatch

All wired in `src/lib.rs::install_menus`, `install_keybindings`, `install_action_handlers`. **Order of those calls matters** — see [Ordering invariant](#ordering-invariant) below.

### Menus (`cx.set_menus`)

- **Eidola**: About / Check for Updates… / Settings… / Hide / Hide Others / Show All / Quit
- **File**: New Space / Library… / Record… / Close Window
- **Edit**: Undo / Redo / Cut / Copy / Paste / Select All — Cut/Copy/Paste/Select All declared via `MenuItem::os_action(_, _, OsAction::*)` so they bind to the standard macOS selectors `cut:` / `copy:` / `paste:` / `selectAll:` and route through the responder chain to whatever has focus
- **Window**: Minimize / Zoom

`cx.set_dock_menu` adds "New Space" and "Library…" for the dock right-click menu.

**The "Window" menu name is special.** gpui_macos detects a menu literally named `"Window"` and calls `app.setWindowsMenu_(menu)` — which is how AppKit recognizes the app as a fully-wired macOS app and reliably dispatches menu key-equivalents in edge cases (no key window after ⌘Tab back, all windows closed). The Hide / Hide Others / Show All trio play the same "I'm a real app" signaling role.

### Keybindings (`cx.bind_keys`)

⌘, (Settings), ⌘N (NewSpace), ⌘L (OpenLibrary), ⇧⌘L (OpenRecord), ⌘W (CloseWindow), ⌘Q (Quit), ⌘H (Hide), ⌥⌘H (HideOthers), ⌘M (Minimize), plus `ChatView`-context bindings: ⌘↩ (`Send`), ⌥⌘M (`ToggleModelPicker` — distinct from global ⌘M), and the picker-navigation set: `escape` → `DismissModelPicker`, `up` → `PickerUp`, `down` → `PickerDown`, `enter` → `PickerConfirm` (all scoped to `ChatView`; action handlers gate on picker-open so they are no-ops when the picker is closed).

### Ordering invariant

`install_keybindings(cx)` **must** run before `install_menus(cx)`. `cx.set_menus` snapshots the keymap when it builds NSMenuItems and attaches each item's `keyEquivalent` from `keymap.bindings_for_action(action)`. Setting menus before binding keys leaves the keymap empty at lookup time, no keystroke is attached, and macOS can't intercept the shortcut at the menu level — which then breaks ⌘N / ⌘Q etc. when no window has key focus (the only path that *requires* the menu-level intercept; with a window focused, gpui's per-window binding dispatch handles it independently). **Diagnostic signal**: items appear in the menu without their shortcut text on the right side.

### Action handlers (`cx.on_action`)

Most handlers are global (registered on `App`). Two notable patterns:

- **Window-targeting handlers** (`Minimize`, `Zoom`) capture `cx.active_window()` and call `cx.defer` to invoke `window.minimize_window()` / `zoom_window()` *after* the current update completes. Without `defer`, a direct `handle.update(cx, …)` on the same window we were dispatched inside fails (its slot is already taken), `.ok()` swallows the Err, and nothing happens.
- **`CloseWindow` is registered per-view**, not globally. Each view does `.on_action(cx.listener(|_, _: &CloseWindow, window, _| window.remove_window()))` on its root v_flex, and `track_focus`es a handle that's `focus()`ed in the view's constructor (so the dispatch path reaches the listener even before the user clicks anything). The intentional consequence: `is_action_available` returns true only when a window with the listener is alive, so macOS auto-disables the "Close Window" menu item (and ⌘W) when no window is open.

### Lifecycle

- `cx.activate(true)` at launch: brings the app to the foreground from frame 0 so the menu bar is fully connected before the user interacts.
- `Application::on_reopen` (registered on the `Application` builder *before* `run()` — the method takes `&self` and returns `&Self`, while `run()` consumes by value, so it can't be chained inline; bind the application to a local first): when the dock icon is clicked with no windows open, opens a new chat window. Without this, closing the last window leaves the app running but unreachable.

## .app bundling — required, not cosmetic

A bare `cargo run -p eidola-gui` binary launches as a command-line tool from AppKit's perspective, not a real app. `setActivationPolicy(Regular)` papers over the common path (menu shows, items enable, mouse clicks dispatch), but **menu key-equivalents fail to dispatch when no window is key**, even with the keymap-ordering fix above. The diagnostic signal is the menu bar showing the binary name (`eidola-gui`) instead of the app name (`Eidola`).

The fix is a proper macOS bundle:

- `crates/eidola-gui/Support/Info.plist` — `CFBundleIdentifier = tech.m6i.eidola-gpui`, `CFBundleExecutable = Eidola`, `NSPrincipalClass = NSApplication`, `NSHighResolutionCapable = true`.
- `scripts/package-gui-app.sh` — copies `target/{debug,release}/eidola-gui` to `Contents/MacOS/Eidola` (renamed to match `CFBundleExecutable` — mismatch falls back to tool-mode), copies the Info.plist, ad-hoc codesigns. Output at `crates/eidola-gui/build/Eidola.app` (gitignored). This is the **dev path** — fast cargo iteration, ad-hoc signed for local launch only.
- `just build gui` runs `cargo build` then the package script on macOS. `just run gui` builds + `open`s the .app. Non-macOS falls back to `cargo run`.
- **Release path:** `nix build .#eidola-gui-macos-universal` (flake.nix) builds an aarch64 + x86_64 universal binary, zeros LC_UUID for reproducibility, and assembles the same `Eidola.app` layout into a Nix store output that's recorded as `eidola-gui-macos-universal` in `artifact-manifest.json` (Nix `narHash`, `darwin/universal` platform). Built locally via `just update-manifest` (alongside the CLI universal binary) and in CI by the `apple` job. Two reproducibility tricks make this work inside the Nix sandbox: (1) gpui's Metal shaders are compiled at app startup via `runtime_shaders` (passed through `gpui_platform/runtime_shaders` → `gpui_macos/runtime_shaders`) instead of via Apple's closed-source `metal` compiler at build time; (2) `flake.nix`'s `commonArgs.preBuild` stitches a missing `assets/assets/icons` sibling layout into the cargo-vendored gpui-component dep, working around an upstream packaging bug where the `icon_named!` proc-macro reads `../assets/assets/icons` relative to its `CARGO_MANIFEST_DIR` (a path that only resolves in the upstream workspace layout, not in `cargo vendor`'s flattened tree).

## Testing — two tiers

Both run via `cargo test -p eidola-gui`. `crates/eidola-gui` has both `[lib]` and `[[bin]]` so the integration tests can import view modules.

### Behavior tests (`tests/behavior.rs`) — the regression gate

Built on `gpui::TestAppContext` (mocked rendering, deterministic dispatcher) so they run on libtest's worker thread without AppKit. Pattern:

1. Build stub stores with `Stores::stub_with(StoresStub { … }, cx)` — a declarative fixture scene (config snapshot, balances, prices, spaces, …).
2. Open a window with the view under test (via `cx.open_window`).
3. Drive interactions through the view's `focus_handle()` — the same path keystrokes take in production.
4. Assert against the view's public state, or the store entities (`stores.account.read_with(cx, …)`), with `read_with`.

Stub stores have no backend, so `Stores::app_core()` returns `None`; views that hit that path early-return after the local state mutation, and store `refresh_*`/`request_*` are no-ops. The store *machinery* itself (supersede, the bus bridge's dispatch, the launch-order replay) is tested against a real tempdir-backed `AppCore` in `tests/stores.rs` (via `Stores::for_test`). HTTP-mocked tests (real `AppCore` against a `wiremock` server) are the natural next layer.

### Visual snapshots (`tests/visual.rs`) — local-only debug aid

**Not** a regression gate. Built on `gpui::VisualTestAppContext` (real Metal renderer, offscreen window at -10000,-10000, deterministic dispatcher). Configured as `[[test]] harness = false` so `fn main()` runs on the macOS main thread (libtest's worker-thread harness would SIGABRT inside AppKit).

Cases live in `tests/visual/cases.rs`; the harness in `tests/visual/harness.rs` wraps each user view in a `Root` and renders it **twice — once in Circadian Day, once in Circadian Night** — by calling `Theme::change` between renders. Each case writes/compares two files: `tests/snapshots/<name>-day.png` and `<name>-night.png`. Case build closures must be `Fn` (invoked once per mode); they construct fresh entities each call.

The PNGs are **gitignored** — pixels are platform- and machine-bound (Metal+CoreText vs wgpu+cosmic-text on Linux; font hinting differs across macOS minor versions), so committing them would mean false-positive regressions in CI and on every other developer's machine. Their value is local: agents/humans can `Read` a PNG to "feel" a view at a state, and a developer iterating on a UI change can re-render and eyeball-diff their previous run.

Behavior:

- Missing PNG → write it and report `written`.
- Mismatch against a previously-written local PNG → write `<name>-<mode>.new.png` for review and fail.
- `UPDATE_SNAPSHOTS=1` overwrites.

Recipes: `just render-snapshots` (verify/write), `just render-snapshots-update` (accept).

### Why both tiers?

Behavior tests catch logic regressions (clicking X must call `store.Y(z)`; an empty Send must be a no-op) and survive across platforms. Visual snapshots are the "did I accidentally change the layout?" check that's only meaningful to the dev making the change. Together they let agents make UI changes confidently: behavior tests gate the merge, visual snapshots let the agent verify the change *looks* right by reading the freshly-written PNG.

## gpui / gpui-component pinning

`gpui-component` (longbridge, rev `dadfca97fec7221acf3ce7047bccdc1eac0506b9`) pulls `gpui` and `gpui_platform` from `zed-industries/zed` without a rev. We mirror that exact spec in `Cargo.toml` so cargo unifies on a single `gpui` copy. `Cargo.lock` is the canonical pin for the resolved zed commit. gpui-component and gpui move in lockstep: a given gpui-component rev expects a matching gpui (e.g. the `flex_shrink_1`/`flex_grow_1` helpers it calls landed in gpui commit `8982fb17`), so bump both together.

**The zed pin is held at `969a67fc`, deliberately *behind* `main`.** The next commit, `39f7849a` ("Log worst hanging tasks and actions", zed #57835), added a global action-timing profiler whose process-wide `ACTION_STATISTICS` spinlock isn't isolated per thread. It's harmless in the single-threaded UI at runtime, but libtest runs our `gpui-markdown-editor` behavior tests in parallel — concurrent `dispatch_action` calls clobber the one global `running` slot and `save_action_timing()` panics on `None`. `969a67fc` is the parent of that commit: it has the flex helpers gpui-component needs but predates the profiler. Because the deps are rev-less, the lock is the only thing holding this — **do not `cargo update` zed past `969a67fc` until the upstream profiler race is fixed** (re-test the editor suite under parallelism before advancing the pin).
